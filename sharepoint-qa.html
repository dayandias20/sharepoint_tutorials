Comprehensive Technical & Scenario-Based Questions for HESAA Position

Technical Questions

1 How would you design the information architecture for HESAA's SharePoint intranet to serve different departments (Financial Aid Programs, Communications, IT, Student Services)?

I'd approach this using a hub site architecture for scalability and governance:

Structure:

  • Primary Hub: HESAA Intranet Home (corporate communications, news, policies)
  • Department Hub Sites:
    • Financial Aid Programs Hub
    • Communications Hub
    • IT Services Hub
    • Student Services Hub

Design Rationale:

  • Hub sites provide shared navigation, branding, and search aggregation while maintaining department autonomy
  • Each department gets its own site collection (not subsites) for independent permissions, storage quotas, and recycle bins
  • Use site templates to ensure consistency while allowing customization
  • Implement term store taxonomy with department-specific term sets for metadata tagging

Navigation Design:

  • Global navigation showing all hubs
  • Local navigation within each hub for department-specific resources
  • Audience targeting to show relevant content based on department membership

Key Features:

  • Centralized document libraries with department-specific folders and permissions
  • Department-specific news web parts aggregated to home page
  • Power Automate workflows for document approvals within departments
  • Search-driven experience with metadata refiners (department, document type, program)
2 Walk me through your process for managing SharePoint permissions when you have sensitive student financial data that needs restricted access.

Security Model:

I'd use SharePoint security groups (not AD groups directly) for granular control:

Tiered Permission Structure:

  1. Site Collection Admins: IT administrators only
  2. Site Owners: Department managers (Financial Aid Director)
  3. Site Members: Standard department staff (read/edit non-sensitive content)
  4. Custom Permission Levels:
    • "Financial Data Reviewers" (View Only - no download)
    • "Financial Data Processors" (Edit specific libraries only)
    • "Compliance Officers" (Full Control on sensitive libraries)

Implementation Process:

  1. Break inheritance at the document library level for sensitive data
  2. Create dedicated security groups for each access level
  3. Use Item-level permissions sparingly (only when necessary for specific cases)
  4. Enable Access Request settings routing to compliance officer
  5. Implement Sensitivity labels (DLP policies) to auto-classify and protect documents
  6. Enable audit logging to track who accesses sensitive files

Best Practices:

  • Document the permission structure in a governance document
  • Regular permission audits quarterly using PowerShell reports
  • Remove "Everyone" and "All Users" groups from sensitive libraries
  • Use Azure AD conditional access to require MFA for sensitive site access
  • Enable versioning and retention policies to prevent data loss
3 Explain the difference between SharePoint Online and SharePoint Server. What are the pros and cons of each for a government agency?
Aspect SharePoint Online (SPO) SharePoint Server (On-Prem)
Hosting Microsoft cloud (Azure) Your own servers/data center
Updates Automatic, continuous updates Manual patching (monthly CU/PU)
Cost Per-user subscription (~$5-20/user/month) Server licenses + infrastructure + IT staff
Scalability Unlimited, cloud-scale Limited by hardware capacity
Integration Native M365 integration (Teams, OneDrive, Power Platform) Limited integration, requires configuration
Customization SPFx framework only (modern) Full server-side code, farm solutions
Security Microsoft manages infrastructure security Full control over security configuration

For a Government Agency (HESAA):

SharePoint Online PROS:

  • ✅ Compliance: FedRAMP certified, meets government standards
  • ✅ Cost-effective: No server hardware, reduces IT overhead
  • ✅ Disaster recovery: Built-in geo-redundancy and backup
  • ✅ Modern features: Latest collaboration tools (Teams, Planner, Power Platform)
  • ✅ Mobile access: Cloud accessibility for remote work
  • ✅ Reduced attack surface: Microsoft handles infrastructure patching

SharePoint Online CONS:

  • ❌ Data sovereignty: Data stored in Microsoft cloud (may require GCC High for sensitive data)
  • ❌ Less customization: Limited to SPFx, no server-side code
  • ❌ Internet dependency: Requires reliable connectivity
  • ❌ Compliance scrutiny: May require approval from state IT security boards

SharePoint Server PROS:

  • ✅ Data control: All data stays on-premises
  • ✅ Full customization: Farm solutions, event receivers, custom code
  • ✅ Hybrid options: Can integrate with SPO gradually
  • ✅ Network isolation: Can be completely air-gapped if required

SharePoint Server CONS:

  • ❌ High cost: Servers, SQL, IT staff for maintenance
  • ❌ Manual patching: Monthly security updates required
  • ❌ Scalability limits: Hardware constraints
  • ❌ Legacy features: Missing modern experiences, Teams integration

Recommendation for HESAA:

SharePoint Online (potentially GCC) - Government agencies are increasingly adopting cloud solutions. HESAA would benefit from cost savings, automatic updates, and modern collaboration tools. If student financial data requires special handling, consider GCC (Government Community Cloud) for enhanced compliance.

4 How do you perform vulnerability assessments and apply security patches to a SharePoint Server environment?

Vulnerability Assessment Process:

1. Monthly Security Review (First Tuesday of Month)

  • Review Microsoft Security Response Center (MSRC) bulletin
  • Check SharePoint-specific CVEs and severity ratings
  • Cross-reference with NIST National Vulnerability Database

2. Scanning Tools

  • Microsoft Baseline Security Analyzer (MBSA): Scan for missing patches
  • Nessus/Qualys: Third-party vulnerability scanning
  • SharePoint Health Analyzer: Built-in health check rules
  • Security & Compliance Center: Review compliance score

3. Assessment Areas

  • Operating system patches (Windows Server)
  • SQL Server patches and service packs
  • SharePoint cumulative updates (CU) or public updates (PU)
  • IIS configuration hardening
  • SSL/TLS certificate validity
  • Firewall rules and network segmentation

Patching Process:

Phase 1: Planning (Day 1-3)

  1. Review patch notes and known issues on SharePoint community forums
  2. Check compatibility with custom solutions (SPFx, farm solutions)
  3. Schedule maintenance window (typically Saturday night)
  4. Notify stakeholders via email and intranet banner

Phase 2: Testing (Day 4-7)

  1. Create VM snapshot of production farm
  2. Deploy patch to test/staging environment first
  3. Run automated tests (smoke tests, critical workflows)
  4. Verify custom web parts and workflows function
  5. Test search crawl and user permissions
  6. Get sign-off from business owners

Phase 3: Production Deployment (Maintenance Window)

  1. Backup everything:
    • SQL content databases (full backup)
    • Configuration database
    • Service application databases
    • IIS configuration export
  2. Stop SharePoint services (Timer Service, Search, User Profile Sync)
  3. Apply patches in order:
    • SQL Server patches first
    • Windows OS patches
    • SharePoint binaries (run as farm admin)
  4. Run PSConfig.exe or SharePoint Products Configuration Wizard
  5. Upgrade content databases (if required)
  6. Start services and verify health

Phase 4: Verification (Post-Patch)

  1. Run Get-SPProduct -Local to verify version
  2. Check SharePoint Health Analyzer for errors
  3. Test critical functionality:
    • Site access and permissions
    • Search functionality
    • Workflow execution
    • Custom web parts
  4. Monitor ULS logs for errors
  5. Send completion notification to stakeholders

Tools Used:

  • PowerShell scripts: Automated patching across farm servers
  • AutoSPInstaller: For consistent farm builds
  • SharePoint Management Shell: Health checks and verification
  • SCCM/WSUS: Centralized patch management
5 Describe your experience with SharePoint migration. What tools and strategies would you use to migrate legacy content?

Migration Experience:

I led a migration of 2TB of content from SharePoint 2013 to SharePoint Online, including 25+ site collections serving 800+ users.

Migration Strategy:

Phase 1: Discovery & Assessment (2-3 weeks)

  1. Inventory current environment:
    • Run Get-SPSite -Limit All to list all site collections
    • Use SharePoint Migration Assessment Tool (SMAT) to identify issues
    • Document custom solutions (workflows, web parts, InfoPath forms)
  2. Content analysis:
    • Identify inactive sites (no activity >1 year)
    • Large files (>250MB limit in SPO)
    • Broken links, orphaned items, duplicate content
  3. Create migration wave plan:
    • Pilot sites (low-risk departments)
    • Production waves (department by department)
    • Sensitive data last (Financial Aid)

Phase 2: Remediation (3-4 weeks)

  1. Clean up content:
    • Archive/delete inactive sites (save 30-40% storage)
    • Break large files into smaller chunks or move to OneDrive
    • Retire SharePoint 2010 workflows → migrate to Power Automate
    • Convert InfoPath forms → PowerApps
  2. Redesign information architecture:
    • Flatten deep folder hierarchies (improve findability)
    • Implement hub sites instead of nested subsites
    • Create modern team sites instead of classic sites

Phase 3: Migration Execution

Tools I'd Use:

1. SharePoint Migration Tool (SPMT) - Microsoft's free tool

  • Best for: Document libraries, lists, small site collections
  • Pros: Free, supports incremental migration, auto-mapping
  • Cons: Limited customization, can be slow for large datasets

2. Sharegate - Third-party (my preferred tool)

  • Best for: Complex migrations, enterprise scale
  • Pros:
    • Excellent permission mapping and reporting
    • Pre-migration health checks
    • Incremental migration (delta sync)
    • Workflow and web part analysis
  • Cons: Licensing cost (~$5K-15K)

3. PowerShell with PnP - For custom scenarios

Connect-PnPOnline -Url "https://source.sharepoint.com/sites/OldSite" Get-PnPListItem -List "Documents" -PageSize 500 Connect-PnPOnline -Url "https://tenant.sharepoint.com/sites/NewSite" Add-PnPFile -Path $file -Folder "Shared Documents"

4. Microsoft 365 Migration App - For file shares and content

  • Good for migrating network drives to SharePoint/OneDrive

Migration Approach:

  1. Pilot migration (1 department, 50 users):
    • Test tool, validate process
    • Train users on modern SharePoint
    • Gather feedback, refine approach
  2. Wave migrations (bi-weekly waves):
    • Migrate during off-hours (Friday night)
    • Run delta sync before cutover
    • Dual-run for 1 week (old + new access)
  3. Post-migration validation:
    • Permission verification (spot-check 20% of files)
    • Search crawl completion
    • Workflow testing
    • User acceptance testing

Common Issues & Solutions:

  • Versioning limits: SPO has version limits → trim old versions pre-migration
  • Path length: 400-character limit → flatten folders
  • Special characters: #, % in file names → rename before migration
  • Customizations: Farm solutions don't work in SPO → redevelop as SPFx
6 How do you monitor and optimize SharePoint site performance when sites are loading slowly?

Troubleshooting Approach:

Step 1: Identify the Bottleneck

Browser-Level Diagnostics:

  1. Open F12 Developer Tools → Network tab
  2. Identify slow-loading resources:
    • Large images (>500KB)
    • Blocking JavaScript files
    • Slow API calls (>2 seconds)
  3. Check console errors (JavaScript exceptions)
  4. Use Lighthouse (Audit tab) for performance score

SharePoint-Level Diagnostics:

  1. Modern pages: Click "Diagnostics" → Download HAR file
  2. Review page weight (target: <2MB total)
  3. Check web part performance (some web parts are resource-intensive)
  4. Review list view threshold (over 5,000 items?)

Server-Level (On-Prem):

  1. Check SharePoint Health Analyzer for warnings
  2. Review ULS logs for slow queries or timeouts
  3. Monitor SQL Server performance:
    • CPU usage, memory pressure
    • Long-running queries (sp_who2)
    • Blocking/deadlocks
  4. Check IIS logs for slow requests (>10 seconds)

Step 2: Common Issues & Fixes

Issue 1: Large List Views (>5,000 items)

  • Symptom: "List view threshold" error or slow rendering
  • Fix:
    • Create indexed columns (modified date, content type)
    • Add filters to views (show last 30 days only)
    • Enable folder-based organization
    • Use modern list instead of classic

Issue 2: Unoptimized Images

  • Symptom: Slow page load, large network transfer
  • Fix:
    • Use SharePoint image renditions (auto-resize)
    • Compress images before upload (TinyPNG, ImageOptim)
    • Use modern image web part (lazy loading)
    • Enable CDN for image libraries (SPO only)

Issue 3: Too Many Web Parts

  • Symptom: Slow initial page render
  • Fix:
    • Limit to 6-8 web parts per page
    • Remove unused web parts
    • Use "See all" links instead of showing 50 items
    • Lazy load web parts (scroll into view)

Issue 4: Custom JavaScript/SPFx Performance

  • Symptom: Slow script execution, browser hangs
  • Fix:
    • Review SPFx bundle size (webpack-bundle-analyzer)
    • Remove unused libraries (tree shaking)
    • Use async/await for API calls
    • Implement caching (localStorage, session storage)
    • Debounce search/filter inputs

Issue 5: Search Crawl Issues (On-Prem)

  • Symptom: Continuous crawling, high CPU
  • Fix:
    • Exclude crawling frequently-changing lists
    • Increase crawl database size
    • Schedule crawls during off-hours
    • Review crawler impact rules

Step 3: Proactive Optimization

SharePoint Online:

  1. Enable SharePoint CDN for static assets:
    Set-SPOTenantCdnEnabled -CdnType Public -Enable $true
  2. Use modern pages (faster than classic)
  3. Implement page caching for static content
  4. Monitor via Microsoft 365 Admin Center → Reports → SharePoint Usage

SharePoint On-Prem:

  1. Distributed Cache: Ensure properly configured (feed cache, security trimming)
  2. BLOB storage: Move large files to RBS (Remote BLOB Storage)
  3. SQL optimization:
    • Regular index maintenance (rebuild/reorganize)
    • Update statistics weekly
    • Partition large databases
  4. Server resources: Scale out web front-ends if CPU >70%

Performance Targets:

  • Page load: <3 seconds (desktop), <5 seconds (mobile)
  • Time to Interactive (TTI): <5 seconds
  • Largest Contentful Paint (LCP): <2.5 seconds
7 What's your approach to SharePoint governance? How do you prevent site sprawl and maintain consistency?

Governance Framework:

1. Site Provisioning Controls

Site Request Process:

  • No self-service site creation for site collections (prevent sprawl)
  • Implement site request form (PowerApps + Power Automate):
    • Department, business justification, site owner
    • Expected number of users, data sensitivity level
    • Approval workflow: Department head → IT → Compliance (if sensitive data)
  • Provision from approved site templates only (ensure branding/structure)

Site Lifecycle Management:

  • Expiration policies: Sites expire after 2 years of inactivity
  • Automated reminders to site owners (90, 60, 30 days before expiration)
  • Recertification: Annual review by site owners to confirm still needed
  • Archival process: Move inactive sites to "Archived" web app (on-prem) or read-only (SPO)

2. Permission Management

Access Control Standards:

  • Principle of least privilege: Users get minimum access needed
  • No "Everyone" group: Use AD security groups for department-based access
  • Break inheritance sparingly: Document all permission breaks in governance log
  • Regular audits: Quarterly permission reviews via PowerShell reports
Get-SPWeb -Site "https://intranet" -Limit All | Where-Object {$_.HasUniqueRoleAssignments} | Select-Object URL, Title

3. Content Organization Standards

Taxonomy & Metadata:

  • Enterprise term store: Centralized taxonomy for:
    • Departments (Financial Aid, IT, Communications)
    • Document types (Policy, Procedure, Form, Report)
    • Program areas (NJCLASS, EOF, TAG, NJ Stars)
  • Mandatory metadata: Require department and document type on libraries
  • Content types: 5-7 standard content types (prevent proliferation)

Naming Conventions:

  • Sites: [Department]-[Purpose] (e.g., "FinAid-Policies", "IT-KnowledgeBase")
  • Libraries: Descriptive names (avoid "Documents", use "Policies", "Forms")
  • Folders: Discourage deep nesting (max 3 levels)

4. Template & Branding Standards

Approved Templates:

  • Department site template (team collaboration)
  • Project site template (temporary projects)
  • Knowledge base template (policies, procedures)
  • Prevent custom site templates (maintenance nightmare)

Design Consistency:

  • Hub site branding: Centrally managed theme, logo, navigation
  • Approved color palette: HESAA brand colors only
  • Modern experience: Disable classic UI where possible
  • Custom master pages: Avoid (breaks upgrades, creates tech debt)

5. Preventing Site Sprawl

Strategies:

  1. Educate site owners: Monthly training on Teams vs. SharePoint vs. OneDrive
    • Teams: Collaboration, chat, meetings
    • SharePoint: Departmental sites, intranet, formal documents
    • OneDrive: Personal files, draft documents
  2. Promote Teams: Many site requests should actually be Teams
  3. Hub sites: Encourage subsites under hubs (not new site collections)
  4. Discovery: Monthly reports on new sites created, review with governance board
  5. Decommissioning: Proactively retire unused sites (auto-archive after 1 year inactivity)

6. Governance Documentation

Essential Documents:

  • SharePoint governance plan (15-20 pages)
  • Site request and provisioning process
  • Permission management standard
  • Naming conventions and taxonomy guide
  • Site owner responsibilities (SOPs)
  • Retention and archival policies

Governance Board:

  • Quarterly meetings: IT, Compliance, Department representatives
  • Review site requests, policy updates, security incidents
  • Approve new content types, templates, customizations

7. Monitoring & Reporting

Monthly Reports:

  • New sites created (review justification)
  • Sites with broken inheritance (permission audits)
  • Largest sites (storage management)
  • Inactive sites (candidates for archival)
  • External sharing activity (security review)

Tools:

  • PowerShell scripts: Automated reporting
  • SPDocKit (third-party): Governance dashboards, permission reports
  • SharePoint Admin Center: Usage analytics, audit logs
8 Explain the difference between hub sites, site collections, and subsites. When would you use each?
Feature Site Collection Subsite Hub Site
Definition Top-level container, independent unit Child site within site collection Virtual grouping of site collections
URL Pattern https://intranet/sites/hr https://intranet/sites/hr/onboarding Multiple site collections associated
Permissions Independent permission boundary Inherits or breaks from parent Each site maintains own permissions
Storage Quota Has own quota (on-prem) Shares parent quota N/A (virtual grouping)
Recycle Bin Two-stage recycle bin Shares parent recycle bin N/A
Search Own search scope Included in parent scope Aggregates search across associated sites
Navigation Independent Can inherit parent Shared navigation across associated sites
Admin Site collection admin Site owner Hub owner configures branding/nav

When to Use Each:

Site Collections - Use for:

  • ✅ Department sites with sensitive data (Financial Aid, HR)
    • Reason: Independent permission boundary, own recycle bin
  • ✅ Large departments needing storage control
    • Reason: Dedicated storage quota (on-prem)
  • ✅ Sites requiring distinct branding (external partner portals)
    • Reason: Own theme, master page (classic)
  • ✅ Long-term sites (not temporary projects)
    • Reason: Easier to manage lifecycle independently

Example:

  • https://hesaa.sharepoint.com/sites/FinancialAid
  • https://hesaa.sharepoint.com/sites/IT
  • https://hesaa.sharepoint.com/sites/Communications

Subsites - Use for:

  • ✅ Closely related sub-teams (HR → Onboarding, Benefits, Recruitment)
    • Reason: Logical hierarchy, shared navigation
  • ✅ Temporary projects within a department
    • Reason: Easy to delete without affecting parent
  • ✅ When you want inherited permissions (most users have same access)
    • Reason: Simplifies permission management

⚠️ WARNING: Microsoft de-emphasizes subsites in modern SharePoint

  • Modern team sites don't support subsites well
  • Harder to migrate, flat structures preferred
  • Use Teams channels instead of subsites for collaboration

Example:

  • https://hesaa.sharepoint.com/sites/HR (parent)
    • https://hesaa.sharepoint.com/sites/HR/Onboarding (subsite)
    • https://hesaa.sharepoint.com/sites/HR/Benefits (subsite)

Hub Sites - Use for:

  • ✅ Connecting related site collections without making subsites
    • Reason: Shared navigation and branding without subsite limitations
  • ✅ Department-wide portals (Financial Aid Hub connecting program sites)
    • Reason: Unified navigation across multiple sites
  • ✅ Roll-up content (aggregate news, events across department sites)
    • Reason: Hub news web part, search roll-ups
  • ✅ Modern intranet architecture (replacing subsites)
    • Reason: Flexible, scalable, modern approach

Example - HESAA Hub Structure:

Hub: Financial Aid Programs (hub site)

Associated sites:

  • NJ CLASS Loans (site collection)
  • EOF Program (site collection)
  • TAG Grants (site collection)
  • NJSTARS Program (site collection)

Benefit: Shared navigation, roll-up news from all programs, unified search

Hub: IT Services (hub site)

Associated sites:

  • Help Desk (site collection)
  • Projects (site collection)
  • Documentation (site collection)

My Recommendation for HESAA:

Use Hub Site Architecture:

HESAA Intranet (Communication Site - Home) ├─ Hub: Financial Aid Programs │ ├─ Associated: NJ CLASS Site (Team Site) │ ├─ Associated: EOF Program (Team Site) │ └─ Associated: TAG Grants (Team Site) ├─ Hub: Communications │ ├─ Associated: News & Events (Communication Site) │ └─ Associated: Marketing Resources (Team Site) ├─ Hub: IT Services │ ├─ Associated: IT Help Desk (Team Site) │ └─ Associated: IT Projects (Team Site) └─ Hub: Student Services └─ Associated: Student Support (Team Site)

Why This Works:

  • Each site collection = independent permissions (critical for HESAA's sensitive data)
  • Hub sites = unified navigation and branding
  • Scalable (easy to add new program sites)
  • Modern architecture (future-proof)
  • No subsites (avoid technical debt)

Scenario-Based Questions

9 HESAA staff report they can't access an important financial aid document library. Walk me through how you would troubleshoot this.

Step-by-Step Troubleshooting:

Phase 1: Gather Information (5 minutes)

  1. Who is affected?
    • Single user, specific department, or everyone?
    • Document user name(s) and department
  2. What exactly happens?
    • "Access Denied" error?
    • Page doesn't load (blank screen, timeout)?
    • Specific documents or entire library?
  3. When did this start?
    • Sudden (last hour) or gradual?
    • After any changes (permission updates, migration, patching)?
  4. Has it ever worked for these users?
    • New employees vs. existing users losing access

Phase 2: Quick Checks (10 minutes)

Test 1: Verify with Your Admin Account

  • Log in with site collection admin credentials
  • Can YOU access the library?
    • ✅ YES → Permission issue (user-specific)
    • ❌ NO → Site/library issue (affects everyone)

Test 2: Check Site Status (SharePoint Online)

  • Open SharePoint Admin Center → Sites → Active sites
  • Is the site showing as "Active" or any warnings?
  • Check Microsoft 365 Service Health for SharePoint outages

Test 3: Test with Another User in Same Department

  • Can their colleagues access it?
    • ✅ YES → User-specific issue
    • ❌ NO → Department group or permission issue

Phase 3: Permission Verification (15 minutes)

Check User Permissions:

  1. Navigate to Library Settings → Permissions for this document library
  2. Click Check Permissions (ribbon)
  3. Enter affected user's email → Click "Check Now"
  4. Review what permission level they have (if any)

Common Permission Issues:

Issue 1: Broken Inheritance Without Adding User

  • Symptom: User had access yesterday, not today
  • Cause: Someone broke inheritance on library but forgot to add users
  • Fix:
    • Library Settings → Permissions → Grant Permissions →
    • Add [Department Group] with "Read" or "Contribute"

Issue 2: User Not in Required AD Security Group

  • Symptom: "Access Denied - You need permission to access this site"
  • Cause: Library uses AD group "Financial_Aid_Staff" but user not member
  • Fix:
    • Check AD group membership: Get-ADGroupMember "Financial_Aid_Staff"
    • Add user to group (may take 24 hours to sync)
    • Immediate workaround: Add user directly to SharePoint group

Issue 3: Item-Level Permissions Blocking Access

  • Symptom: Can see library, but specific documents show "Access Denied"
  • Cause: Someone set unique permissions on folders/files
  • Fix:
    • Go to specific document → "..." → Details → Manage Access
    • Check "Who has access" → Add user if missing

Phase 4: Technical Issues (20 minutes)

Issue 4: Site Collection Lock State (On-Prem)

Get-SPSite "https://intranet/sites/FinancialAid" | Select-Object LockState
  • If ReadOnly or NoAccess → Site was locked (storage quota or admin lock)
  • Fix: Set-SPSite -Identity [URL] -LockState "Unlock"

Issue 5: Authentication Issues

  • Symptom: Prompted for credentials repeatedly, never loads
  • Cause: Kerberos misconfiguration, expired SSL cert, browser cache
  • Fix:
    • Clear browser cache: Ctrl+Shift+Delete
    • Try incognito/private mode
    • Try different browser (Edge vs. Chrome)
    • Check if user can access from different computer

Issue 6: Limited Access Lockdown Mode (Classic Sites)

  • Symptom: Users with "Limited Access" can't view library even though they should
  • Cause: Feature ViewFormPagesLockDown is activated
  • Fix:
Disable-SPFeature -Identity "ViewFormPagesLockDown" -Url "https://intranet/sites/FinancialAid"

Phase 5: Advanced Diagnostics (30 minutes)

Check Audit Logs (SharePoint Online):

  1. Microsoft Purview Compliance Center → Audit → Search
  2. Filter:
    • Activities: "File and page activities"
    • Users: [Affected user]
    • Site: [Financial Aid site URL]
  3. Look for "Access denied" events showing what triggered denial

Check ULS Logs (On-Prem):

Get-SPLogEvent -StartTime (Get-Date).AddHours(-2) | Where-Object {$_.Message -like "*Access Denied*"} | Select-Object Timestamp, Message

Network/Firewall Issues:

  • Can user access other SharePoint sites?
  • If off-site, is VPN connected?
  • Check firewall rules blocking SharePoint URLs

Phase 6: Resolution & Communication

Immediate Workaround:

  • If urgent access needed and permission fix takes time:
    • Create SharePoint group "Financial_Aid_Temp_Access"
    • Add affected users
    • Grant group access to library
    • Remove once permanent fix deployed

Root Cause Fix:

  • Document what caused issue (broken inheritance, missing group)
  • Implement fix (restore inheritance, add group membership)
  • Test with affected users
  • Verify access working

Communication:

  1. Acknowledge: "I've received your report and am investigating" (within 15 min)
  2. Update: "I've identified the issue - permission group was removed. Fixing now" (30 min)
  3. Resolution: "Access restored. Please test and confirm. Root cause: [explain]" (1 hour)
  4. Prevention: "I've documented this and will audit permissions quarterly to prevent recurrence"

Post-Incident:

  • Update runbook/documentation
  • If recurring issue → implement monitoring alert
  • Review governance (why did permissions break?)
10 You need to restructure the SharePoint intranet to improve findability of resources. How would you approach this project?

Project Approach:

Phase 1: Discovery & Research (2-3 weeks)

1. Understand Current Problems

  • User interviews (15-20 staff across departments):
    • "How do you find documents today?"
    • "What resources do you struggle to find?"
    • "How often do you use search vs. browse?"
  • Analytics review:
    • Top searched keywords (failed searches = findability problems)
    • Most visited pages (what's working)
    • Least visited pages (orphaned content)
  • Heuristic evaluation:
    • Count clicks to reach common resources (goal: ≤3 clicks)
    • Test navigation labels (jargon vs. plain language)
    • Identify broken links, outdated content

2. Content Inventory

  • Catalog all sites, libraries, pages (use SharePoint Auditor tool)
  • Document content types:
    • Policies (40 documents)
    • Procedures (60 documents)
    • Forms (200 documents)
    • Program resources (150 documents)
  • Identify duplicates, outdated content (haven't been modified in 2+ years)

3. User Research

  • Card sorting exercise (20 participants):
    • Give users 50 content items on cards
    • Ask them to group and label categories
    • Reveals how users mentally organize information
  • Tree testing:
    • Present proposed navigation structure
    • Ask users to find specific items
    • Measure success rate (goal: >80%)

Phase 2: Information Architecture Design (2 weeks)

1. Organize Content by User Tasks (Not Departments)

Before (Department-Based - Poor Findability):

├─ Financial Aid Department ├─ IT Department ├─ Communications Department └─ Student Services Department
