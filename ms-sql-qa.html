MS SQL Server Quick Reference

Essential Concepts for HESAA Web Developer Position

1. Fundamentals

What is MS SQL Server?

Microsoft SQL Server is a relational database management system (RDBMS) developed by Microsoft. It stores and retrieves data as requested by other software applications.

Core Components

  • Database: Container for tables, views, stored procedures, and other objects
  • Table: Structured collection of data organized in rows and columns
  • Row (Record): Single entry in a table
  • Column (Field): Attribute of data with a specific data type

Common Data Types

Type Description Example
INT Integer numbers 123, -456
VARCHAR(n) Variable-length string (up to n characters) 'John Doe'
NVARCHAR(n) Unicode variable-length string 'José García'
DECIMAL(p,s) Fixed precision and scale numbers 123.45
DATE Date values '2025-12-07'
DATETIME Date and time values '2025-12-07 14:30:00'
BIT Boolean (0 or 1) 0, 1

Keys & Constraints

  • Primary Key: Uniquely identifies each row in a table (cannot be NULL)
  • Foreign Key: Links two tables together, references primary key of another table
  • UNIQUE: Ensures all values in a column are different
  • NOT NULL: Ensures a column cannot have NULL values
  • CHECK: Ensures values in a column meet a specific condition
  • DEFAULT: Sets a default value for a column when no value is specified

2. Basic SQL Operations (CRUD)

CREATE - Creating Tables

CREATE TABLE Students ( StudentID INT PRIMARY KEY IDENTITY(1,1), FirstName NVARCHAR(50) NOT NULL, LastName NVARCHAR(50) NOT NULL, Email NVARCHAR(100) UNIQUE, EnrollmentDate DATE DEFAULT GETDATE(), GPA DECIMAL(3,2) CHECK (GPA >= 0.0 AND GPA <= 4.0) );

INSERT - Adding Data

-- Insert single row INSERT INTO Students (FirstName, LastName, Email, GPA) VALUES ('John', 'Doe', 'john.doe@example.com', 3.75); -- Insert multiple rows INSERT INTO Students (FirstName, LastName, Email, GPA) VALUES ('Jane', 'Smith', 'jane.smith@example.com', 3.85), ('Bob', 'Johnson', 'bob.j@example.com', 3.50);

SELECT - Retrieving Data

-- Select all columns SELECT * FROM Students; -- Select specific columns SELECT FirstName, LastName, GPA FROM Students; -- Select with WHERE clause SELECT * FROM Students WHERE GPA >= 3.5; -- Select with ORDER BY SELECT * FROM Students ORDER BY GPA DESC; -- Select with LIKE (pattern matching) SELECT * FROM Students WHERE Email LIKE '%@example.com';

UPDATE - Modifying Data

UPDATE Students SET GPA = 3.90 WHERE StudentID = 1; -- Update multiple columns UPDATE Students SET FirstName = 'Jonathan', Email = 'jonathan.doe@example.com' WHERE StudentID = 1;

DELETE - Removing Data

-- Delete specific rows DELETE FROM Students WHERE StudentID = 5; -- Delete all rows (use with caution!) DELETE FROM Students;
⚠️ Important: Always use WHERE clause with UPDATE and DELETE to avoid modifying/deleting all rows unintentionally!

3. JOINs - Combining Data from Multiple Tables

Understanding JOINs

JOINs combine rows from two or more tables based on a related column between them.

INNER JOIN

Returns only matching rows from both tables.

SELECT Students.FirstName, Students.LastName, Courses.CourseName FROM Students INNER JOIN Enrollments ON Students.StudentID = Enrollments.StudentID INNER JOIN Courses ON Enrollments.CourseID = Courses.CourseID;

LEFT JOIN (LEFT OUTER JOIN)

Returns all rows from the left table, and matching rows from the right table. NULL for non-matching right table rows.

-- Get all students, including those not enrolled in courses SELECT Students.FirstName, Students.LastName, Courses.CourseName FROM Students LEFT JOIN Enrollments ON Students.StudentID = Enrollments.StudentID LEFT JOIN Courses ON Enrollments.CourseID = Courses.CourseID;

RIGHT JOIN (RIGHT OUTER JOIN)

Returns all rows from the right table, and matching rows from the left table.

SELECT Students.FirstName, Courses.CourseName FROM Students RIGHT JOIN Enrollments ON Students.StudentID = Enrollments.StudentID;

FULL OUTER JOIN

Returns all rows from both tables, with NULL where there's no match.

SELECT Students.FirstName, Courses.CourseName FROM Students FULL OUTER JOIN Enrollments ON Students.StudentID = Enrollments.StudentID;

CROSS JOIN

Returns the Cartesian product of both tables (all possible combinations).

SELECT Students.FirstName, Courses.CourseName FROM Students CROSS JOIN Courses;

JOIN Visual Summary

  • INNER JOIN: Only matching records ∩
  • LEFT JOIN: All from left + matching from right (Left table ∪ Match)
  • RIGHT JOIN: All from right + matching from left (Right table ∪ Match)
  • FULL OUTER JOIN: All from both tables (Left ∪ Right)

4. Aggregation & Grouping

Aggregate Functions

-- COUNT: Number of rows SELECT COUNT(*) AS TotalStudents FROM Students; -- AVG: Average value SELECT AVG(GPA) AS AverageGPA FROM Students; -- SUM: Total sum SELECT SUM(Credits) AS TotalCredits FROM Courses; -- MIN and MAX SELECT MIN(GPA) AS LowestGPA, MAX(GPA) AS HighestGPA FROM Students;

GROUP BY

Groups rows that have the same values in specified columns.

-- Count students by enrollment year SELECT YEAR(EnrollmentDate) AS Year, COUNT(*) AS StudentCount FROM Students GROUP BY YEAR(EnrollmentDate); -- Average GPA by major SELECT Major, AVG(GPA) AS AvgGPA FROM Students GROUP BY Major;

HAVING

Filters groups (used with GROUP BY). WHERE filters rows before grouping, HAVING filters after grouping.

-- Find majors with average GPA > 3.5 SELECT Major, AVG(GPA) AS AvgGPA FROM Students GROUP BY Major HAVING AVG(GPA) > 3.5;

5. ACID Properties

ACID ensures database transactions are processed reliably:

A - Atomicity

All or Nothing: A transaction either completes entirely or not at all. If any part fails, the entire transaction is rolled back.

Example: Transferring $100 from Account A to Account B requires two operations: deduct from A, add to B. If adding to B fails, the deduction from A must be rolled back.

C - Consistency

Valid State: A transaction brings the database from one valid state to another, maintaining all defined rules (constraints, triggers, cascades).

Example: If a CHECK constraint requires GPA between 0.0 and 4.0, no transaction can insert a GPA of 5.0.

I - Isolation

Concurrent Execution: Concurrent transactions are isolated from each other. Intermediate state of one transaction is invisible to others.

Example: If two users update the same student record simultaneously, one transaction must wait for the other to complete to avoid conflicts.

D - Durability

Permanent Changes: Once a transaction is committed, it remains committed even in case of system failure (crash, power loss).

Example: After committing a student's grade, that grade is permanently stored and will survive server restarts or crashes.

6. Transactions

Transaction Control

BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1; UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2; -- If all operations succeed COMMIT TRANSACTION; -- If any operation fails ROLLBACK TRANSACTION;

Error Handling with TRY...CATCH

BEGIN TRY BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1; UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2; COMMIT TRANSACTION; END TRY BEGIN CATCH ROLLBACK TRANSACTION; PRINT 'Transaction failed: ' + ERROR_MESSAGE(); END CATCH;

7. Indexes

What are Indexes?

Indexes improve query performance by creating a data structure that allows faster data retrieval. Think of it like a book's index—instead of reading every page to find a topic, you check the index.

Clustered Index

  • Determines the physical order of data in the table
  • One per table (usually on primary key)
  • Faster for range queries
CREATE CLUSTERED INDEX IX_Students_StudentID ON Students(StudentID);

Non-Clustered Index

  • Separate structure from table data
  • Multiple per table (up to 999 in SQL Server)
  • Faster for specific column lookups
CREATE NONCLUSTERED INDEX IX_Students_Email ON Students(Email);
When to Index: Columns frequently used in WHERE, JOIN, ORDER BY clauses. Avoid over-indexing as it slows down INSERT/UPDATE/DELETE operations.

8. Views

What are Views?

A view is a virtual table based on a SELECT query. It doesn't store data itself but provides a way to simplify complex queries.

-- Create a view showing student enrollment summary CREATE VIEW vw_StudentEnrollmentSummary AS SELECT s.StudentID, s.FirstName + ' ' + s.LastName AS FullName, COUNT(e.CourseID) AS CourseCount, AVG(c.Credits) AS AvgCredits FROM Students s LEFT JOIN Enrollments e ON s.StudentID = e.StudentID LEFT JOIN Courses c ON e.CourseID = c.CourseID GROUP BY s.StudentID, s.FirstName, s.LastName; -- Query the view like a table SELECT * FROM vw_StudentEnrollmentSummary WHERE CourseCount > 3;

Benefits of Views

  • Simplification: Hide complex JOINs and business logic
  • Security: Expose only specific columns/rows to users
  • Abstraction: Insulate applications from schema changes

9. Stored Procedures

What are Stored Procedures?

Precompiled SQL code stored in the database that can accept parameters and return results.

-- Create a stored procedure to enroll a student CREATE PROCEDURE sp_EnrollStudent @StudentID INT, @CourseID INT, @EnrollmentDate DATE AS BEGIN BEGIN TRY INSERT INTO Enrollments (StudentID, CourseID, EnrollmentDate) VALUES (@StudentID, @CourseID, @EnrollmentDate); PRINT 'Student enrolled successfully.'; END TRY BEGIN CATCH PRINT 'Enrollment failed: ' + ERROR_MESSAGE(); END CATCH END; -- Execute the stored procedure EXEC sp_EnrollStudent @StudentID = 1, @CourseID = 101, @EnrollmentDate = '2025-01-15';

Benefits

  • Performance: Precompiled execution plans
  • Security: Control access to data through procedures
  • Reusability: Encapsulate business logic
  • Reduced network traffic: Execute multiple statements in one call

10. Normalization

What is Normalization?

The process of organizing data to reduce redundancy and improve data integrity.

Normal Forms

1NF (First Normal Form)

  • Eliminate repeating groups
  • Each column contains atomic (indivisible) values
  • Each column has a unique name
  • Order doesn't matter

2NF (Second Normal Form)

  • Must be in 1NF
  • All non-key columns fully depend on the primary key
  • Eliminate partial dependencies

3NF (Third Normal Form)

  • Must be in 2NF
  • No transitive dependencies (non-key columns don't depend on other non-key columns)
  • All columns depend only on the primary key

Denormalized (Bad):

StudentID | StudentName | CourseID | CourseName | InstructorName

1 | John Doe | 101 | Math | Dr. Smith

1 | John Doe | 102 | English | Dr. Jones

Problem: Student name repeats, causes update anomalies

Normalized (Good):

Students: StudentID | StudentName

Courses: CourseID | CourseName | InstructorName

Enrollments: EnrollmentID | StudentID | CourseID

11. Performance Optimization

Query Optimization Tips

  • Use indexes strategically on frequently queried columns
  • Avoid SELECT *—specify only needed columns
  • Use WHERE instead of HAVING when filtering before grouping
  • Avoid functions in WHERE clauses on indexed columns (breaks index usage)
  • Use EXISTS instead of IN for large subqueries
  • Limit result sets with TOP or OFFSET/FETCH

Execution Plans

View how SQL Server executes a query to identify bottlenecks:

-- Show execution plan SET SHOWPLAN_TEXT ON; GO SELECT * FROM Students WHERE GPA > 3.5; GO SET SHOWPLAN_TEXT OFF;

Common Performance Issues

  • Table scans: Add indexes to avoid full table scans
  • Missing statistics: Update statistics regularly
  • Blocking/deadlocks: Optimize transaction isolation levels
  • Parameter sniffing: Use query hints or RECOMPILE

12. Advanced Concepts

Window Functions (ROW_NUMBER, RANK, DENSE_RANK)

-- Rank students by GPA SELECT FirstName, LastName, GPA, ROW_NUMBER() OVER(ORDER BY GPA DESC) AS RowNum, RANK() OVER(ORDER BY GPA DESC) AS Rank, DENSE_RANK() OVER(ORDER BY GPA DESC) AS DenseRank FROM Students;

Common Table Expressions (CTEs)

-- Temporary named result set WITH HighPerformers AS ( SELECT StudentID, FirstName, LastName, GPA FROM Students WHERE GPA >= 3.5 ) SELECT * FROM HighPerformers ORDER BY GPA DESC;

Subqueries

-- Students with GPA above average SELECT FirstName, LastName, GPA FROM Students WHERE GPA > (SELECT AVG(GPA) FROM Students);

PIVOT and UNPIVOT

-- Pivot enrollments by semester SELECT * FROM ( SELECT StudentID, Semester, Credits FROM Enrollments ) AS SourceTable PIVOT ( SUM(Credits) FOR Semester IN ([Fall], [Spring], [Summer]) ) AS PivotTable;

13. HESAA-Relevant Scenarios

Student Financial Aid Database Example

Scenario: Query students eligible for specific grants

SELECT s.StudentID, s.FirstName + ' ' + s.LastName AS FullName, a.AwardName, a.AwardAmount, a.AwardDate FROM Students s INNER JOIN FinancialAidApplications faa ON s.StudentID = faa.StudentID INNER JOIN Awards a ON faa.ApplicationID = a.ApplicationID WHERE faa.ApplicationStatus = 'Approved' AND a.AwardType = 'Grant' AND a.AwardDate >= '2025-01-01' ORDER BY a.AwardAmount DESC;

Scenario: Track application processing time

SELECT ApplicationID, StudentID, SubmissionDate, DecisionDate, DATEDIFF(DAY, SubmissionDate, DecisionDate) AS ProcessingDays FROM FinancialAidApplications WHERE ApplicationStatus = 'Approved' ORDER BY ProcessingDays DESC;

Scenario: Generate monthly disbursement report

SELECT YEAR(DisbursementDate) AS Year, MONTH(DisbursementDate) AS Month, ProgramType, COUNT(*) AS DisbursementCount, SUM(DisbursementAmount) AS TotalAmount FROM Disbursements GROUP BY YEAR(DisbursementDate), MONTH(DisbursementDate), ProgramType HAVING SUM(DisbursementAmount) > 10000 ORDER BY Year, Month;
