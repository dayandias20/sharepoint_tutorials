Accessibility & ADA Compliance

Interview Questions & Expert Answers for HESAA Position

Technical Questions

28 How do you ensure a website meets WCAG 2.1 Level AA standards? Walk me through your testing process.

Four-phase approach:

  1. Automated scanning with axe DevTools/WAVE to catch obvious issues (alt text, contrast)
  2. Keyboard testing—navigate entire site using Tab/Enter only
  3. Screen reader testing with JAWS/NVDA to verify announcements and labels
  4. Manual code review for semantic HTML and proper ARIA

Document findings by severity and track fixes.

29 Explain the importance of ARIA labels and when you would use them. Give specific examples.

ARIA adds meaning when native HTML isn't enough. Use when:

  • aria-label="Close" on a <button>×</button> to describe visual-only icons
  • aria-describedby="error-msg" to link error text to invalid fields
  • aria-expanded="true/false" on accordions to announce state
  • aria-live="polite" for dynamic content updates

Never override native semantics—prefer <button> over <div role="button">.

30 How do you make complex data tables (like financial aid program comparisons) accessible to screen reader users?

Use proper markup:

  • <caption> for table description
  • <th scope="col/row"> for headers
  • Connect headers to cells with id/headers attributes for complex tables

For very complex tables, provide a simplified view or convert to card layout on mobile.

Example:
<th scope="col" id="eligibility"> paired with
<td headers="eligibility program-njclass">
31 What tools do you use for accessibility testing? (JAWS, NVDA, axe DevTools, etc.)

Screen Readers: JAWS, NVDA, VoiceOver

Browser Tools: axe DevTools, WAVE, Lighthouse

Color Contrast: WebAIM Contrast Checker

Automation: Pa11y for CI/CD

Automated tools catch ~30-40% of issues; manual testing catches the rest (tab order, link clarity, context).

32 How do you ensure PDFs posted on the HESAA website are accessible?

Create accessible source documents (proper headings, alt text) before PDF conversion.

In Adobe Acrobat Pro:

  • Run Accessibility Checker
  • Tag document structure
  • Set reading order
  • Add alt text
  • Ensure color contrast (4.5:1)
  • Make forms fillable with labels
  • Run OCR on scans

Always provide HTML alternative when possible.

33 Explain how you would test keyboard-only navigation on a website.

Disconnect mouse. Use Tab to move forward, Shift+Tab backward—verify all interactive elements are reachable.

Check:

  • Visible focus indicators on each element
  • Enter for links/buttons, Space for checkboxes
  • Arrow keys for dropdowns
  • Escape to close modals
  • Skip links work
  • Focus doesn't trap in modals
34 What are common accessibility issues in SharePoint, and how do you remediate them?
Issue: Poor color contrast → Fix: Custom WCAG-compliant theme
Issue: Missing form labels → Fix: Add <label> or aria-label
Issue: Unlabeled images → Fix: Require alt text, train authors
Issue: Complex navigation → Fix: Hub sites, skip links, proper heading hierarchy
Issue: Non-accessible PDFs → Fix: Train on Adobe Acrobat Pro remediation

Scenario-Based Questions

35 An accessibility audit reveals 50+ WCAG violations on HESAA's website. How would you prioritize and address these?

Critical (Week 1): Forms unusable by keyboard, missing form labels, text contrast <3:1

High (1 Month): Missing alt text, broken heading structure, keyboard traps

Medium (3 Months): Contrast 3-4.49:1, PDF issues, ARIA improvements

Start with quick wins (theme changes affecting whole site), fix highest-traffic pages first, train content authors, implement governance checklist.

36 A user with low vision complains they can't read text on the website. How would you investigate and fix this?

Investigate:

  • Check color contrast (needs 4.5:1)
  • Font size (minimum 16px)
  • Zoom to 200% without breaking
  • Test High Contrast Mode

Fix:

  • Darken text/lighten background for contrast
  • Increase base font to 16px with relative units (rem)
  • Use responsive design for zoom
  • Provide high-contrast theme toggle
  • Ensure 44px touch targets

Key Takeaways

  • ✅ Accessibility is non-negotiable for government agencies
  • ✅ Automated tools catch 30-40%; manual testing essential
  • ✅ WCAG 2.1 Level AA is the standard (4.5:1 contrast, keyboard access, screen reader support)
  • ✅ Train content authors on accessibility best practices
  • ✅ Test with real assistive technology (JAWS, NVDA, keyboard-only)
