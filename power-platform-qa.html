Technical Questions

11 Design a Power Automate flow for HESAA's grant application approval process with multiple approval stages and conditional routing.

I'd use a sequential approval flow with conditional logic. The flow triggers when a grant application is submitted to a SharePoint list, then routes through three stages:

Stage 1: Program Manager approval
Stage 2: Financial Officer approval (if amount >$10K)
Stage 3: Director final approval

Each stage uses the Approval action with timeout settings (5 business days). If rejected at any stage, the flow sends a notification to the applicant and updates the application status. I'd use parallel branches for notifications and scope actions to organize stages, making the flow easier to maintain.

12 How do you handle errors and failures in Power Automate workflows? Give specific examples.

I use Try-Catch patterns with Scope actions: wrap risky operations in a "Try" scope, then add a parallel "Catch" scope configured to run only "if Try has failed."

Example: Document Approval Flow

If the "Send Email" action fails due to invalid recipient, the Catch scope:

  • Logs the error to a SharePoint list
  • Notifies the admin
  • Sets the item status to "Error - Needs Review"

I also enable run-after configuration on critical actions and implement retry policies (3 attempts with exponential backoff) for API calls that might timeout.

13 Explain how you would create a workflow that routes student assistance applications to different departments based on program type.

I'd create a condition-based routing flow triggered when an application is submitted. Use a Switch action on the "Program Type" field with cases for each program:

NJ CLASS → Financial Aid Team
EOF → Student Services
TAG → Grant Processing

Each branch then:

  1. Assigns the item to the appropriate SharePoint group
  2. Sends a customized notification email to that department's queue
  3. Creates a task in their department's Planner board

I'd include a default case for unknown program types that routes to a general intake team for manual review.

14 How do you test and debug complex Power Automate flows before deploying to production?

I follow a three-tier testing approach:

1. Development

Test with sample data using the "Test" button, checking each action's inputs/outputs in the run history.

2. Staging

Deploy to a test SharePoint site with real-like data and have 2-3 users perform end-to-end testing.

3. User Acceptance Testing

Have business owners test critical scenarios.

Debugging Techniques:

  • Use Compose actions as breakpoints to inspect variable values
  • Enable detailed logging by adding "Log to SharePoint" actions at key stages
  • Use Parallel branches to send debug emails showing current flow state
  • Check the 28-day run history for patterns in failures
15 Describe your experience with scheduled flows versus event-driven flows. When do you use each?

Event-Driven Flows

Triggered by: SharePoint item creation, email arrival, Teams message

Best for: Real-time processes like approval workflows or notifications—they execute immediately when something happens.

Scheduled Flows

Triggered by: Recurrence (e.g., daily, weekly)

Best for: Batch operations like nightly report generation, weekly inactive site cleanup, or daily deadline reminders.

Example from Experience:

At my university, I used:

  • Event-driven flows for grant submission approvals (instant response)
  • Scheduled flows to check expiring documents every Monday at 8 AM and send reminder emails

Scheduled flows are also better for API rate-limiting scenarios where you need to process items in controlled batches.

16 How would you create a workflow that automatically notifies stakeholders when deadlines are approaching?

I'd create a scheduled flow that runs daily at 7 AM:

Step 1: Query Items
Uses Get Items with filter:
DueDate le addDays(utcNow(), 7) and Status ne 'Completed'
(finds items due within 7 days)
Step 2: Loop Through Results
Uses Apply to each to iterate through results
Step 3: Calculate Days Remaining
Uses addDays and dateDifference expressions
Step 4: Send Tiered Notifications
Uses Switch/Condition for urgency levels:
  • 7 days: Informational email
  • 3 days: High-priority email + Teams notification
  • 1 day: Urgent email to stakeholder + their manager

I'd store "Last Notification Sent" in SharePoint to avoid duplicate reminders.

18 A critical workflow is failing intermittently. How do you troubleshoot and fix it?

Troubleshooting Approach:

Step 1: Identify the Pattern (15 min)

  • Check Flow Run History for the last 20 runs—look for common failure points
  • Note if failures happen at specific times (timeouts during high-load periods?) or with specific data (special characters breaking filters?)
  • Export failed run details to Excel and analyze patterns

Step 2: Examine Error Messages (15 min)

  • Drill into failed runs and identify the exact action failing
  • Common culprits:
    • API timeouts (429 errors)
    • Permission issues (401/403)
    • Null reference errors (missing data)
    • Concurrency conflicts

Step 3: Reproduce & Debug (30 min)

  • Use Test mode with the same data that caused failures
  • Add Compose actions before/after the failing action to inspect input/output values
  • Check for issues like:
    • Incorrect filter syntax
    • Missing null checks (if(empty(triggerOutputs()?['body/field'])...))
    • Exceeding SharePoint threshold (5000 items)

Step 4: Implement Fix

If Timeout: Add retry policy (3 attempts, exponential backoff) or split into smaller batches
If Permission: Verify connection credentials and SharePoint permissions
If Data Issue: Add validation/null checks before the action
If Concurrency: Enable flow concurrency control or add delay actions

Step 5: Monitor Post-Fix (1 week)

  • Enable email notifications for flow failures
  • Review run history daily for 3 days, then weekly
  • Document the issue and fix in runbook for future reference

Example Fix:

Problem: A grant approval flow was failing intermittently. Root cause: when applicants left the "Department" field blank, the flow couldn't route correctly.

Solution: Added a condition to check if Department is null—if yes, default to "General Intake" queue and notify applicant to update their application.
